# Page Seek


> 一个聚焦于**论文检索知识库与问答**的 RAG 平台，支持论文文档接入、索引构建、会话记忆、多通道检索、流式问答与可视化运营。

## 项目定位

Page Seek 的核心定位不是通用聊天机器人，而是一个围绕**论文资料沉淀、学术内容检索、基于证据的问答生成**构建的知识平台。

它适合用于：

- 论文 PDF / Markdown / URL 等资料的统一接入与管理
- 研究主题、方法、实验结论、指标定义等内容的结构化检索
- 基于知识库上下文的论文问答、综述辅助与阅读辅助
- 面向研究场景的多轮对话，保留会话上下文与摘要记忆
- 管理后台下的知识库、文档、Chunk、检索链路与问答追踪

一句话概括：**Page Seek = 面向论文场景的检索知识库 + 问答系统 + 记忆增强 RAG 平台。**

## 核心能力

- **论文知识库**：支持按知识库组织论文、报告、笔记等学术资料
- **索引构建流程**：支持文档解析、切块、增强、富化、向量化、入库的完整流水线
- **多通道检索**：支持全局向量检索与意图定向检索并行执行，再进行去重与重排
- **会话记忆机制**：支持历史轮次保留 + 摘要压缩，兼顾长对话连续性与上下文成本
- **流式问答体验**：SSE 实时输出答案，适合论文问答与深度阅读场景
- **意图识别与路由**：区分知识检索、系统回复、工具调用、歧义引导等不同处理路径
- **可观测性**：支持 RAG Trace、检索链路追踪、Chunk 与文档管理
- **扩展能力**：支持 MCP Server，对接实时工具或外部数据源

## 项目流程

从“论文进入系统”到“用户得到答案”，整体流程可以概括为两条主链路。

### 1）论文入库与索引构建

1. 创建论文知识库，配置 Embedding 模型与向量集合
2. 上传论文文件，或通过 URL / 远程地址导入文档
3. 进入摄取流水线：`Fetcher -> Parser -> Chunker -> Enhancer -> Enricher -> Indexer`
4. 解析出正文与结构化内容，按策略切分为 Chunk
5. 对 Chunk 做标题补全、上下文增强、元数据富化等处理
6. 生成向量并写入向量库，同时保留文档与 Chunk 元数据
7. 后台支持查看分块结果、分块日志、索引状态与文档状态

### 2）论文问答与检索生成

1. 用户发起问题，请求进入流式对话管道
2. 系统先加载当前会话记忆，并把本轮问题写入记忆
3. 对用户问题做查询改写、上下文补全与多子问题拆分
4. 解析问题意图，判断是否需要歧义引导或系统直答
5. 若进入知识问答，则调用多通道检索引擎
6. 检索结果经过去重、重排、聚合后进入 Prompt 编排
7. LLM 基于“问题 + 记忆 + 检索上下文”生成最终答案并流式返回

## 索引构建设计

对于论文场景，索引质量直接决定问答上限，因此 Page Seek 在索引构建上不是简单的“切块 + 向量化”，而是强调**可配置、可追踪、可增强**。

### 流水线节点

项目内置的摄取引擎支持按节点串联执行，每个节点都可以独立配置、记录日志并输出结果：

- `FetcherNode`：获取原始文档内容
- `ParserNode`：解析 PDF / 文本 / 远程资源，提取结构化文档
- `ChunkerNode`：按策略切分段落、章节或语义片段
- `EnhancerNode`：增强 Chunk 内容，改善检索召回表达
- `EnricherNode`：补充元数据，如来源、任务、节点输出信息
- `IndexerNode`：校验向量维度、确保向量空间存在，并写入向量库

### 为什么这套索引更适合论文检索

- **比固定切块更稳**：论文常见长段落、表述密集、章节结构明显，流水线可以按场景调优切块策略
- **比直接入向量库更强**：增强与富化环节能为 Chunk 补足标题、来源与上下文，让语义检索更准确
- **比黑盒导入更可控**：每个节点都有执行日志，便于排查“为什么这篇论文没检索到”
- **比单一索引更可运营**：文档、Chunk、向量集合、分块日志都能在后台管理

如果将本项目用于论文库建设，可以重点围绕以下维度调优：

- Chunk 大小与重叠策略
- 标题 / 摘要 / 章节名注入方式
- 元数据字段设计，如作者、年份、研究方向、关键词、数据集、方法名
- Embedding 模型与集合划分方式

## 会话记忆机制

论文问答往往不是单轮检索，而是连续追问，例如：

- “这篇论文的核心贡献是什么？”
- “它和上一段提到的方法有什么区别？”
- “实验部分里提升最大的指标是哪一个？”

如果没有记忆机制，后续问题里的“它”“上一段”“这个方法”都很难被正确理解。Page Seek 在问答链路里加入了**短期记忆 + 摘要压缩**机制。

### 记忆加载方式

系统在每次对话开始时，会并行加载：

- 最近若干轮原始消息
- 当前会话最新摘要

随后将“摘要 + 最近历史”一起拼装到模型上下文里，让模型既保留近期细节，也能理解更早的讨论背景。

### 记忆压缩方式

当会话轮数达到阈值后，系统会在回答完成后异步触发摘要压缩：

- 以最近保留轮次为边界，截取更早历史
- 将历史摘要与新增历史一起送入 LLM 合并
- 生成受字符上限约束的新摘要
- 以系统消息形式参与后续 Prompt 构建

这种设计的好处是：

- **保留连续追问能力**：适合论文精读、方法对比、实验追踪
- **控制上下文成本**：不会因为多轮追问导致 Prompt 无限膨胀
- **兼顾稳定性**：近期原文保留，远期信息摘要化
- **便于工程落地**：消息与摘要都持久化在数据库中，摘要生成带分布式锁保护

## 检索与生成链路

Page Seek 的问答流程不是单一的“向量检索后直接拼 Prompt”，而是做了更完整的编排。

### 查询改写

针对论文问答中的省略、指代和上下文依赖，系统会先做查询改写，包括：

- 指代补全
- 上下文补足
- 多子问题拆分

这一步的目标是让“适合人类表达的问题”，转换成“更适合检索系统理解的问题”。

### 多通道检索

当前检索引擎支持两类核心通道：

- **全局向量检索**：面向整个知识库集合召回相关 Chunk
- **意图定向检索**：基于识别出的意图节点，在更聚焦的范围内召回结果

通道会并行执行，随后再统一进入后处理阶段。

### 后处理与生成

检索结果会经过：

- 去重
- 重排
- 上下文聚合
- Prompt 场景化编排

最后根据场景选择对应 Prompt 模板，组合：

- 系统提示词
- 会话记忆
- 论文知识上下文
- 用户当前问题

由模型生成最终答案，并以 SSE 流式返回。

## 后台与运维能力

项目提供论文知识库场景下常用的运营后台能力：

- 知识库管理
- 文档管理
- Chunk 管理
- 索引任务与流水线配置
- 问答链路追踪（RAG Trace）
- 系统设置、模型配置与用户管理

这使得项目不仅能做 Demo，也具备继续扩展成平台化系统的基础。

## 技术栈

### 后端

- Java 17
- Spring Boot 3.5.x
- MyBatis-Plus
- Redisson
- RocketMQ
- Milvus 2.6.x
- Apache Tika

### 前端

- React 18
- TypeScript
- Vite
- Tailwind CSS

### 模块划分

- `bootstrap`：业务入口与核心应用编排
- `framework`：通用基础设施、约定与 Trace 能力
- `infra-ai`：Embedding、Rerank、LLM、Token 等 AI 基础能力
- `mcp-server`：MCP 工具服务与协议适配
- `frontend`：Web 前端与管理后台

## 快速启动

### 1. 基础环境

- JDK 17+
- Maven 3.9+
- Node.js 18+
- PostgreSQL
- Redis
- Milvus
- RocketMQ

### 2. 启动后端

```bash
./mvnw spring-boot:run -pl bootstrap
```

默认后端地址为 `http://localhost:9090/api/page-seek`。

### 3. 启动前端

```bash
cd frontend
npm install
npm run dev
```

前端开发服务器默认通过 Vite 启动，本地代理可转发到后端接口。

### 4. 准备论文知识库

建议按以下顺序体验：

1. 创建知识库
2. 上传论文文档
3. 执行分块与索引构建
4. 在问答页围绕论文内容发起多轮问题
5. 在后台查看 Chunk、Trace 与检索结果
